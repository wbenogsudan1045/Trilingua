# Requirements Document

## Introduction

This feature adds Supabase Storage and Database integration to the Trilingua translation application. Translated documents are uploaded to a private Supabase Storage bucket and made available via 7-day signed download URLs. Every document translation job is logged to a `translation_history` table in Supabase Database. A History page lets users view and re-download their past translations within the same browser session.

User authentication is out of scope. The Laravel session ID is used as the user identifier — history is scoped to the current browser session. Text-only translations (no file attachment) are out of scope for storage and history logging.

## Glossary

- **Storage_Service**: The new Laravel service class responsible for uploading translated files to Supabase Storage and generating signed download URLs.
- **History_Service**: The new Laravel service class responsible for writing and reading records in the `translation_history` Supabase table.
- **Translation_Controller**: The existing Laravel controller that orchestrates document and text translation requests.
- **History_Controller**: The new Laravel controller that handles requests to the history page and the re-download endpoint.
- **Supabase_Storage**: The private Supabase Storage bucket that holds translated output files.
- **Supabase_DB**: The Supabase PostgreSQL database that stores the `translation_history` table.
- **Signed_URL**: A time-limited, pre-authenticated URL generated by Supabase Storage that allows a file to be downloaded without additional credentials.
- **Translation_Job**: A single completed document translation, represented as one row in `translation_history`.
- **Session_ID**: The Laravel session ID for the current browser session, used as the user identifier in storage paths and history records.

## Requirements

### Requirement 1: Upload Translated File to Supabase Storage

**User Story:** As a user, I want my translated document stored in the cloud, so that I can retrieve it later without relying on a temporary local file.

#### Acceptance Criteria

1. WHEN a document translation completes successfully, THE Storage_Service SHALL upload the translated output file to Supabase_Storage under a path structured as `{session_id}/{uuid}_translated.{ext}`, where `{ext}` matches the extension of the translated output file.
2. WHEN the upload to Supabase_Storage succeeds, THE Storage_Service SHALL generate a Signed_URL with an expiry of exactly 604 800 seconds (7 days) from the time of generation.
3. WHEN the upload to Supabase_Storage succeeds, THE Storage_Service SHALL return the storage path, the Signed_URL, and the expiry timestamp to the caller.
4. IF the upload to Supabase_Storage fails, THEN THE Storage_Service SHALL throw a descriptive exception containing the Supabase error message, and THE Translation_Controller SHALL return HTTP 500 with a user-facing error message.
5. IF the Signed_URL generation fails after a successful upload, THEN THE Storage_Service SHALL throw a descriptive exception, and THE Translation_Controller SHALL return HTTP 500 with a user-facing error message.
6. WHEN the upload to Supabase_Storage succeeds, THE Translation_Controller SHALL delete the local temporary file from `storage/app/temp/`.

---

### Requirement 2: Log Translation Job to Supabase Database

**User Story:** As a user, I want each of my document translations recorded, so that I have a history of my work within my session.

#### Acceptance Criteria

1. WHEN a document translation and upload both complete successfully, THE History_Service SHALL insert one row into the `translation_history` table in Supabase_DB containing: `session_id`, `original_filename`, `translated_filename`, `source_language`, `target_language`, `created_at` (UTC timestamp), `storage_path`, and `signed_url_expires_at` (UTC timestamp exactly 604 800 seconds after `created_at`).
2. THE `translation_history` table schema SHALL reject any INSERT that omits or sets NULL for any of the columns listed in criterion 1, returning a constraint violation error.
3. IF the insert into `translation_history` fails, THEN THE History_Service SHALL write the error details to the application log and THE Translation_Controller SHALL still return the Signed_URL to the user in the HTTP response so the download is not blocked.

---

### Requirement 3: Return Signed URL in Translation Response

**User Story:** As a user, I want to download my translated document immediately after translation via a cloud link, so that I receive my file without delay.

#### Acceptance Criteria

1. WHEN a document translation completes and the file is uploaded successfully, THE Translation_Controller SHALL return HTTP 200 with a JSON body containing: `download_url` (the Signed_URL), `download_filename` (the user-facing filename), and `signed_url_expires_at` (ISO 8601 UTC timestamp).
2. WHEN the Blade frontend receives a response containing `download_url`, THE frontend SHALL render a download anchor element whose `href` is the `download_url` value and whose `download` attribute is the `download_filename` value.
3. IF the upload or Signed_URL generation fails, THE Translation_Controller SHALL return HTTP 500 and THE frontend SHALL display an error message in the output panel error area without rendering a download link.

---

### Requirement 4: Translation History Page

**User Story:** As a user, I want a dedicated page listing my past document translations from this session, so that I can review and re-download previous work.

#### Acceptance Criteria

1. WHEN a user navigates to the history page, THE History_Controller SHALL query `translation_history` in Supabase_DB for all rows where `session_id` matches the current Laravel session ID, ordered by `created_at` descending, returning at most 200 rows.
2. WHEN the history page has records to display, THE History_Controller SHALL render a Blade view that shows for each Translation_Job: original filename, translated filename, source language, target language, `created_at` formatted as `YYYY-MM-DD HH:MM UTC`, and a "Re-download" button.
3. WHEN the history page contains no records for the current session, THE History_Controller SHALL render the same Blade view with the empty-state message: "You have no translation history yet."
4. WHEN the history page fails to load due to a database error, THE History_Controller SHALL log the exception and render an error view with the message "Unable to load history. Please try again later." without exposing internal error details.

---

### Requirement 5: Re-download via Regenerated Signed URL

**User Story:** As a user, I want to re-download a past translated document at any time, so that I am not limited to the original 7-day window.

#### Acceptance Criteria

1. WHEN a user clicks the "Re-download" button for a Translation_Job whose `session_id` matches the current Laravel session ID, THE History_Controller SHALL call THE Storage_Service to generate a new Signed_URL for the corresponding `storage_path` with an expiry of exactly 604 800 seconds.
2. WHEN the new Signed_URL is generated, THE History_Controller SHALL return HTTP 200 with a JSON body containing the new `download_url`.
3. WHEN the frontend receives the new `download_url`, THE frontend SHALL redirect the user's browser to that URL to initiate the download.
4. IF the file no longer exists in Supabase_Storage, THEN THE History_Controller SHALL return HTTP 404 with the message "This file is no longer available."
5. IF the Storage_Service returns an error other than file-not-found, THEN THE History_Controller SHALL return HTTP 500 with the message "Unable to generate download link. Please try again later."
6. WHEN a new Signed_URL is generated for a Translation_Job, THE History_Service SHALL update the `signed_url_expires_at` column for that row to the new expiry timestamp.
7. IF a re-download request is received for a `translation_history` row whose `session_id` does not match the current Laravel session ID, THEN THE History_Controller SHALL return HTTP 403 without revealing whether the record or file exists.

---

### Requirement 6: Supabase Configuration

**User Story:** As a developer, I want all Supabase credentials managed through environment variables, so that secrets are not committed to source control.

#### Acceptance Criteria

1. THE Laravel application SHALL read the Supabase project URL, anon key, service role key, and storage bucket name exclusively from the environment variables `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, and `SUPABASE_BUCKET`.
2. IF any of the four environment variables listed in criterion 1 is absent or set to an empty string at application boot, THEN THE Laravel application SHALL throw a `RuntimeException` with a message that names the missing variable.
3. THE `.env.example` file SHALL contain a non-empty placeholder value for each of the four environment variables listed in criterion 1 (e.g., `SUPABASE_URL=https://your-project.supabase.co`).
